# AWS 배포 개념 정리
---

## 배포란?

개발한 애플리케이션 코드를 실제 사용자가 접근할 수 있는 서버 환경에 올려놓는 전체 과정을 말한다.
로컬에서 동작하는 코드를 **인터넷에서 누구나 사용할 수 있는 상태**로 만드는 것이 핵심이다.

---

## 1. AWS 인프라 기본 구조

### 리전(Region)

AWS 데이터센터가 모여 있는 지리적 영역이다.

- 서울 리전: `ap-northeast-2`
- 미국 동부: `us-east-1`
- 사용자와 가까운 리전을 선택하면 **지연 시간(latency)** 이 줄어든다.

### 가용 영역(Availability Zone, AZ)

하나의 리전 안에 있는 **독립된 데이터센터 묶음**이다.

- 서울 리전에는 4개의 AZ가 존재한다.
- 하나의 AZ에 장애가 발생해도 다른 AZ에서 서비스를 유지할 수 있다.

### 엣지 로케이션(Edge Location)

CloudFront 같은 CDN이 동작하는 캐시 서버 위치다.
전 세계 수백 개가 분포해 있어 **정적 콘텐츠를 빠르게 전달**한다.

---

## 2. 컴퓨팅 — 어디서 코드를 실행할 것인가?

| 서비스 | 유형 | 특징 |
|---|---|---|
| **EC2** | IaaS (가상 서버) | 가장 기본. OS부터 직접 관리. 자유도 높음 |
| **Elastic Beanstalk** | PaaS | 코드만 업로드하면 EC2 + LB + Auto Scaling 자동 구성 |
| **Lambda** | Serverless | 함수 단위 실행. 요청 시에만 동작. ms 단위 과금 |
| **ECS** | 컨테이너 오케스트레이션 | AWS 자체 컨테이너 관리 서비스 |
| **EKS** | 컨테이너 오케스트레이션 | 관리형 Kubernetes 서비스 |
| **Fargate** | 서버리스 컨테이너 | ECS/EKS에서 서버 관리 없이 컨테이너 실행 |

### EC2 핵심 개념

- **인스턴스 타입**: CPU·메모리 사양 선택 (t2.micro, m5.large 등)
- **AMI (Amazon Machine Image)**: OS와 소프트웨어 구성이 담긴 이미지
- 가장 자유도가 높지만 직접 관리할 것이 많다.

### Lambda 핵심 개념

- 서버 없이 함수 단위로 코드를 실행하는 **서버리스** 서비스
- 항상 떠 있는 서버가 필요 없는 **이벤트 기반 작업**에 적합
- 요청이 올 때만 실행되고, 실행 시간만큼만 과금

---

## 3. 네트워킹 — 트래픽을 어떻게 관리할 것인가?

### VPC (Virtual Private Cloud)

AWS 안에 만드는 **나만의 가상 네트워크**다.

- IP 대역을 정하고, 서브넷을 나누고, 트래픽 허용 규칙을 제어한다.
- 모든 AWS 리소스는 VPC 안에 존재한다.

### 서브넷(Subnet)

VPC를 더 작은 네트워크로 나눈 것이다.

- **퍼블릭 서브넷**: 인터넷에서 직접 접근 가능 → 웹 서버
- **프라이빗 서브넷**: 외부에서 직접 접근 불가 → 데이터베이스

### 보안 그룹(Security Group)

인스턴스 단위의 **방화벽**이다.

```
예: "80번 포트로 들어오는 HTTP 트래픽만 허용"
```

### ELB (Elastic Load Balancer)

들어오는 트래픽을 **여러 서버에 분산**시킨다.

- **ALB (Application LB)**: HTTP/HTTPS 레벨 라우팅
- **NLB (Network LB)**: TCP 레벨 초고속 분산

### Route 53

AWS의 **DNS 서비스**다. 도메인 이름(예: `myapp.com`)을 서버 IP나 로드밸런서에 연결한다.

### CloudFront

**CDN 서비스**다. 전 세계 엣지 로케이션에 콘텐츠를 캐싱해서 사용자에게 빠르게 전달한다.

---

## 4. 스토리지 & 데이터베이스

### S3 (Simple Storage Service)

파일(객체)을 저장하는 서비스다.

- 정적 웹사이트 호스팅, 이미지/동영상 저장, 로그 보관
- **버킷(Bucket)** 이라는 컨테이너 안에 파일을 저장하는 구조

### RDS (Relational Database Service)

MySQL, PostgreSQL, Oracle 등 **관계형 DB 관리형 서비스**다.

- 백업, 패치, 복제를 AWS가 자동으로 처리한다.

### DynamoDB

AWS의 **NoSQL 데이터베이스**다.

- 키-값 구조로 초고속 읽기/쓰기 가능
- 서버리스로 운영

## 5. CI/CD — 코드를 어떻게 자동으로 배포할 것인가?

CI/CD는 **코드 변경 → 빌드 → 테스트 → 배포**를 자동화하는 파이프라인이다.

```
코드 푸시 → CodeCommit → CodeBuild → CodeDeploy
              (저장소)       (빌드/테스트)   (배포)
                    ↑                            ↓
                    └──── CodePipeline 으로 연결 ──┘
```

| 서비스 | 역할 |
|---|---|
| **CodeCommit** | AWS의 Git 저장소 (GitHub과 유사) |
| **CodeBuild** | 소스 코드 컴파일 + 테스트 실행 |
| **CodeDeploy** | 빌드 결과물을 EC2, Lambda, ECS 등에 배포 |
| **CodePipeline** | 위 서비스를 하나의 파이프라인으로 오케스트레이션 |

> 💡 실무에서는 GitHub Actions + AWS 조합을 많이 사용하기도 한다.

## 6. 배포 전략

### 롤링 배포 (Rolling)

서버를 **하나씩 순차적으로** 새 버전으로 교체한다.
배포 중 구버전과 신버전이 공존하는 시간이 있다.

### 블루/그린 배포 (Blue/Green)

현재 운영 환경(블루)과 동일한 새 환경(그린)을 만들고, **트래픽을 한 번에 전환**한다.
문제가 생기면 블루로 되돌리면 되므로 **롤백이 쉽다**.

### 카나리 배포 (Canary)

트래픽의 일부(예: 5%)만 먼저 새 버전으로 보내 **문제가 없는지 확인**한 후 점진적으로 확대한다.

### 인플레이스 (In-place)

기존 서버에서 직접 코드를 교체한다.
가장 단순하지만 **배포 중 다운타임이 발생**할 수 있다.

---

## 7. IaC (Infrastructure as Code)

인프라를 **코드로 정의하고 관리**하는 접근 방식이다.

| 도구 | 특징 |
|---|---|
| **CloudFormation** | AWS 전용. YAML/JSON 템플릿으로 인프라 선언. 스택(Stack) 단위 관리 |
| **Terraform** | HashiCorp 오픈소스. 멀티 클라우드 지원. HCL 문법 사용 |
| **CDK** | CloudFormation을 TypeScript, Python 등 프로그래밍 언어로 작성 |

---

## 8. 모니터링 & 운영

### CloudWatch

CPU 사용률, 메모리, 요청 수 등의 **지표를 수집**하고 **알림을 설정**한다. 로그 확인도 가능하다.

### Auto Scaling

트래픽이 늘면 서버를 **자동으로 추가**하고, 줄면 **자동으로 제거**한다.
CloudWatch 지표를 기반으로 동작한다.

### IAM (Identity and Access Management)

**"누가 어떤 리소스에 어떤 작업을 할 수 있는가"** 를 제어하는 권한 관리 서비스다.
배포 파이프라인에도 적절한 IAM 역할(Role)을 부여해야 한다.

## 9. 전형적인 웹 앱 배포 아키텍처

```
사용자
  │
  ▼
Route 53 (DNS)
  │
  ▼
CloudFront (CDN) ──── S3 (정적 파일)
  │
  ▼
ALB (로드밸런서)
  │
  ▼
EC2 / ECS (애플리케이션)
  │
  ▼
RDS / DynamoDB (데이터베이스)
```
